# 文件操作规范

## 修改文件前

1. **关键代码检查**（如果修改核心逻辑）
   - 搜索目标文件中的 `@关键代码` 标记
   - 查看 `.trae/关键代码清单.md`
   - 如果有关键代码，必须先理解原因再修改
   - 修改后必须验证功能正常
   - 参考 `.trae/代码修改检查清单.md` 完成完整检查

2. 先查阅 `02_项目地图.md`，确认文件的功能定位
3. 检查文件是否已有 `// @审计已完成` 标记
4. 确认修改后文件行数不超过 100 行

## 删除文件前

1. **关键代码检查**（必须首先执行）
   - 在目标文件中搜索 `@关键代码` 标记
   - 查看 `.trae/关键代码清单.md`
   - 如果有关键代码标记，**停止！** 先理解原因
   - 参考 `.trae/代码修改检查清单.md` 完成完整检查

2. 全局搜索确认无其他文件引用
3. 无引用的文件：物理删除
4. 同步更新 `02_项目地图.md` 的【归档区】

## 创建文件时

1. 必须包含中文头部注释
2. 必须添加 `// @审计已完成` 标记
3. 行数控制在 100 行以内
4. 遵循三层架构拆分原则
5. **无论来源**：从外部复制、参考他人代码、AI 生成等任何来源的文件，进入项目文件夹后必须遵守以上规则
6. **新旧代码处理**：同个板块有新代码产生时，必须询问用户是否删除旧代码，避免僵尸代码

## 命名规范

- 文件名：中文拼音（如 `ShuJiTiShiCiPeiZhi.ts`）
- 函数名：中文（如 `huoQuPeiZhi`）
- 变量名：中文（如 `shuMing`）
- 注释：中文

## 行数限制

- 单文件最大 100 行
- 超过 100 行必须拆分
- 拆分架构：数据逻辑层 + 子组件层 + 主视图层
- **自动化检查**：使用 `npm run check:lines` 检查文件行数

## 预防机制

### 文件创建后检查

1. **立即检查行数**：新文件创建后，确认行数<100
2. **运行检查脚本**：`npm run check:lines`
3. **超标立即拆分**：发现超标立即按三层架构拆分

### 任务闭环验证

每个任务完成后，必须执行：
1. `npm run build` - 构建验证
2. `npm run check:lines` - 行数验证
3. 功能测试 - 浏览器验证
4. 文档同步 - 更新项目地图
5. Git 提交 - 避免文件丢失

### 检查脚本

```bash
# 检查所有文件行数
npm run check:lines

# 构建项目
npm run build
```
