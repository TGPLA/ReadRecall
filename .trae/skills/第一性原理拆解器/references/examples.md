
# 第一性原理示例 - Artem的世界

## 示例1：TeddySnaps照片系统

### 表面问题
"我们需要向家长发送更多照片"

### 质疑的假设
1. ❌ 家长想要**更多**照片 → 实际上：他们想要**正确**的照片
2. ❌ 所有时刻都是平等的 → 实际上：有**他们**孩子的时刻才重要
3. ❌ 员工应该策划 → 实际上：员工时间昂贵且不一致

### 基本真理
- 家长想要证明他们特定的孩子是安全/快乐的
- 面孔 = 身份 = "那是**我的**孩子"
- 时间 = 背景 = "这是今天发生的"
- 自动 > 手动以确保一致性

### 重建的解决方案
AI人脸识别 + 自动标记 + 家长专属推送
员工自由拍照 → 系统自动分类和推送

## 示例2：TISA入学流程

### 表面问题
"我们的入学流程太慢"

### 质疑的假设
1. ❌ 我们需要所有信息 upfront → 实际上：我们需要足够的信息来做决定
2. ❌ 表格必须全面 → 实际上：表格造成摩擦
3. ❌ 家长想要提供信息 → 实际上：家长想要**收到**确认

### 基本真理
- 入学 = 承诺的交换（家长承诺孩子，学校承诺名额）
- 最小可行信息：孩子姓名、年龄、家长联系方式
- 其他一切可以稍后补充
- 响应速度 = 质量感知

### 重建的解决方案
第1阶段：姓名 + 年龄 + 邮箱（30秒）→ 即时名额预留
第2阶段：详细表格（在承诺之后）
结果：速度快10倍，转化率更高

## 示例3：GolfTab点餐系统

### 表面问题
"高尔夫球手需要在球场上点餐"

### 质疑的假设
1. ❌ 需要完整的餐厅菜单 → 实际上：简化菜单效果更好
2. ❌ 需要精确的配送时间 → 实际上："在第X洞前"就够了
3. ❌ 复杂的支付流程 → 实际上：记账到房间/标签，稍后结算

### 基本真理
- 高尔夫球手的手经常很脏/戴着手套
- 球场上的注意力 = 最小
- 位置 = 洞号（已经通过开球时间知道）
- 速度 > 选项

### 重建的解决方案
5步点餐：商品 → 数量 → 洞号 → 确认 → 完成
无需浏览菜单，无支付摩擦，无地址输入

## 示例4：员工排班（TeddyKids）

### 表面问题
"员工排班复杂且耗时"

### 质疑的假设
1. ❌ 经理必须创建排班表 → 实际上：AI可以起草，经理批准
2. ❌ 员工偏好不可预测 → 实际上：模式会随时间显现
3. ❌ 比例合规是手动的 → 实际上：这是纯数学

### 基本真理
- 儿童：员工比例是法律规定的（确定性的）
- 员工可用性有规律（可学习的）
- 成本优化是数学的
- 只需要人工监督例外情况

### 重建的解决方案
算法生成排班表 → 经理审查例外 → 员工确认
80%自动化，20%人工判断
