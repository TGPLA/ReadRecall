# 模式优化护栏 (Project-Centric)

在处理用户指令前，必须评估当前模式是否能为该项目提供最优的执行质量。

## 1. 评估标准（基于项目收益）

- **推荐使用 Solo 模式**：
  - 理由：由于任务集中在单一逻辑块，Solo 模式能提供更精准的局部改动，避免不必要的全局扫描干扰，确保代码微调的极速反馈与精准控制。
- **推荐使用 Builder 模式**：
  - 理由：由于任务涉及跨文件联动，Builder 模式能确保整个项目的上下文同步，自动处理接口定义与实现的一致性，防止手动修改多个文件时产生遗漏或破坏架构。

## 2. 拦截与询问逻辑

如果检测到模式与任务不匹配，**必须立即停止执行代码生成**，并按照以下格式回复：

### 场景 A：在 [Solo 模式] 下处理 [跨文件/系统级] 任务

1. **停止执行**：严禁在当前对话框输出任何代码实现。
2. **给出理由**：告知用户："为了确保项目整体架构的一致性并防止跨文件调用出现断层，建议将此任务切换至 **Builder 模式** 执行。Builder 模式能自动为您管理多文件间的依赖同步。"
3. **询问确认**：询问用户："您是否希望切换到 Builder 模式以获得更完整、更安全的自动化支持？或者坚持在当前模式下进行手动局部处理？"

### 场景 B：在 [Builder 模式] 下处理 [极简/孤立] 任务

1. **停止执行**：不要启动 Task Plan 扫描过程。
2. **给出理由**：告知用户："当前任务属于局部逻辑打磨，使用 **Solo 模式** 或 `Cmd+I` 内联编辑能提供更即时的代码反馈和专注度，避免触发复杂的全局扫描流程。"
3. **询问确认**：询问用户："您是否希望切换到更轻量的 Solo 模式进行快速微调？还是继续在此模式下执行？"

## 3. 核心禁令

- **严禁越位**：在获得用户明确指令（如"继续"或手动切换模式）之前，不得输出任何具体的技术方案或代码片段。
- **禁用资源话术**：禁止提及"节省配额"、"Token 消耗"或"计算资源"。所有建议必须从代码质量、开发效率和项目健壮性出发。
