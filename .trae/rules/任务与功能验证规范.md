# 任务与功能验证规范

## 核心原则

**任务完成 ≠ 代码写完** —— 验证 → 文档 → 提交 → 确认

---

## 一、四大铁律

### 1. 立即验证

```
□ npm run build 构建成功
□ npm run dev 启动无报错
□ 浏览器控制台无错误
□ 功能实际可用
□ npm run check:lines 文件行数<100
```

### 2. 文档同步

- 代码变更 → 立即更新文档
- 任务状态 → 验证后再标记
- 目录结构 → 与实际保持一致

### 3. 引用验证

```
创建组件 → import 引用 → JSX 使用 → 验证渲染
```

禁止"孤儿组件"。

### 4. 立即提交

```powershell
git add . && git commit -m "描述" && git status
```

---

## 二、功能五要素验证

| 要素 | 检查方式 |
|------|---------|
| **状态定义** | 搜索状态变量名 |
| **业务逻辑** | 检查状态使用处 |
| **UI 控制入口** | 浏览器查看界面 |
| **交互闭环** | 实际点击测试 |
| **持久化** | 刷新后验证 |

### 组件引用规范

1. **主动追踪数据流**：创建 Hook 时立即搜索 View 层组件，完成 Props 透传
2. **占位符防御**：UI 未实现时留 `// TODO: UI control for [功能名]`
3. **引用验证**：底层组件修改后向上回溯至页面级，确保链路未断

---

## 三、复盘机制

**连续 3 次修复失败** → 停止尝试 → 调用"失败复盘存档"技能

---

## 四、失败案例

### EPUB 右键菜单遗漏

| 检查项 | 状态 |
|--------|------|
| 状态定义 | ✅ |
| 业务逻辑 | ✅ |
| UI 控制入口 | ❌ |
| 用户可操作 | ❌ |
| 持久化 | ✅ |

**教训**：代码存在 ≠ 功能可用
