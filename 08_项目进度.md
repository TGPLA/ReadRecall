### EPUB 滚轮翻页闪烁问题（2026-04-01）

**问题描述**：滚轮翻页时页面出现白屏闪烁，快速滚动时尤为明显。用户反馈为"内容跳动，没有出现平移翻页动画"。

**根本原因：epubjs 底层渲染机制**

epubjs 在翻页时会**先清除旧内容，再渲染新内容**，这是 iframe 渲染的固有行为，无法从应用层改变。白屏发生在旧内容清除后、新内容渲染前的间隙。

**已尝试但失败的方案**：
1. ❌ 添加 500ms 节流 - 节流只能减少触发频率，无法阻止单次翻页时的白屏
2. ❌ 翻页锁 + locationChanged 解锁 - locationChanged 触发时机早于渲染完成
3. ❌ 翻页锁 + rendered 解锁 - rendered 事件触发时白屏已经发生
4. ❌ CSS 淡出/淡入过渡动画 - 淡出后 rendered 事件中的 container 引用是闭包旧值，导致无法淡入，白屏持续
5. ❌ 移除淡出，只保留翻页锁 - 翻页锁无法阻止单次翻页时的白屏

**建议尝试的新方向**：
- **加载遮罩层**：翻页前显示遮罩，渲染完成后隐藏，彻底避免白屏
- **双缓冲预渲染**：预渲染下一页，翻页时直接切换（实现复杂）
- **禁用滚轮翻页**：只保留按钮翻页，避免快速触发

**保留火种（可复用代码）**：
```typescript
// 翻页锁机制（防止连续触发）
const isTurningPageRef = useRef(false);

const handleWheel = (event: WheelEvent) => {
  if (isTurningPageRef.current) return;
  isTurningPageRef.current = true;
  // 翻页...
};

rendition.on('rendered', () => {
  isTurningPageRef.current = false;
});
```

### EPUB 滚轮翻页闭包陷阱问题（2026-04-01）- 已解决

**问题描述**：滚轮翻页时页面一直在第一章内循环，无法跳转到第二章。调试日志显示 `isAtStart` 始终为 `true`，导致每次翻页都执行"从封面跳转到第一章"的逻辑。

**根本原因：JavaScript 闭包陷阱**

在 `useEPUBReaderShiJian.ts` 中，滚轮事件处理器 `handleWheel` 在 `handleRendition` 的 `rendition.hooks.content.register` 回调中定义。这个回调只执行一次（因为 `iframeDoc._wheelHandlerAdded` 标志阻止重复添加）。

**闭包陷阱链条**：
1. `handleNextPage` 是 `useCallback`，依赖 `isAtStart`
2. 当 `setIsAtStart(false)` 被调用后，状态确实更新了
3. React 创建了新的 `handleNextPage` 函数（闭包捕获新的 `isAtStart = false`）
4. **但是** `handleWheel` 中引用的是 `fanYeHeYeMa.handleNextPage`，这是闭包创建时的旧引用
5. `iframeDoc._wheelHandlerAdded = true` 阻止了事件监听器的更新
6. 结果：滚轮事件永远调用的是旧版本的 `handleNextPage`，其 `isAtStart` 永远是 `true`

**已尝试但失败的方案**：
1. ❌ 在 iframe document 上添加 wheel 事件监听 - 正确方向，但未考虑闭包问题
2. ❌ 修复 React Hooks 顺序问题 - 解决了 Hooks 错误，但未解决核心问题
3. ❌ 添加 300ms 节流和 `_wheelHandlerAdded` 标志 - 节流正确，但标志导致闭包无法更新
4. ❌ 添加 spine items 追踪，检测章节末尾 - 逻辑正确，但 `isAtStart` 始终为 true 导致无法执行
5. ❌ 添加调试日志 - 发现 `isAtStart` 始终为 true，确认闭包问题

**建议尝试的新方向**：
- 使用 `useRef` 存储 `handleNextPage` 的最新引用
- 在 `handleWheel` 中通过 `ref.current` 调用，确保每次都获取最新函数
- 或者移除 `_wheelHandlerAdded` 标志，每次都重新绑定事件（需要正确清理旧监听器）

**保留火种（可复用代码）**：
```typescript

rendition.book.loaded.spine.then((spine: any) => {
  spineItemsRef.current = spine.items || [];

// 章节末尾检测逻辑（正确）
if (location?.atEnd && spineItemsRef.current.length > 0) {
  const currentIndex = huoQuDangQianSpineSuoYin();
  if (currentIndex >= 0 && currentIndex < spineItemsRef.current.length - 1) {
    const nextItem = spineItemsRef.current[currentIndex + 1];
```

### 高亮删除后视觉高亮不消失问题（2026-04-03）

**问题描述**：删除高亮记录后，章节中的视觉高亮没有立刻消失，需要刷新页面才能消失。

**根本原因分析**：

问题出在 `useHuaXianChuTi` Hook 中的高亮应用逻辑。`highlights` 状态使用 `useLocalStorageState` 管理，当删除高亮时，状态确实更新了，但 `useEffect` 中绑定的事件回调使用的是闭包中的旧值。

**已尝试但失败的方案**：

| 尝试次数 | 方案描述 | 失败原因 |
|----------|----------|----------|
| 第1次 | 只在 handleDeleteHighlight 中删除 localStorage 数据 | 只删除了数据，没有同步更新章节中的视觉高亮 |
| 第2次 | useEffect 依赖 [highlights, renditionRef]，highlights 变化时重新应用 | useEffect 可能不会在 highlights 变化时立即触发，或者闭包捕获的是旧值 |
| 第3次 | useEffect 不依赖 highlights，改为直接读取 localStorage | 依赖 rendered 事件触发，但删除操作后事件未触发或时机不对 |
| 第4次 | 使用 setTimeout 延迟调用 rendition.annotations.remove(cfiRange, 'highlight') | 视觉高亮仍未消失，cfiRange 可能已失效或格式不匹配 |
| 第5次 | 参考其他项目使用 annotations.remove(cfi, type) + rendition.display() | rendition.display() 导致页面跳转，无法使用 |
| 第6次 | requestAnimationFrame 延迟重新应用高亮 | 视觉高亮仍未消失 |
| 第7次 | 查找 iframe DOM 操作 - .epubjs-hl、data-cfi 等选择器 | iframe 内找不到任何高亮相关 DOM 元素，epubjs 可能使用虚拟渲染 |
| 第8次 | 通过 rendition.themes.default() 刷新样式 | 视觉高亮仍未消失 |
| 第9次 | 通过 rendition.resize() 强制重绘 | 视觉高亮仍未消失 |

**核心矛盾分析**：

经过9轮尝试，发现问题的根本原因可能是：epubjs 0.3.93 版本的高亮注解系统与新版本 API 不同，我们的 `annotations.highlight()` 调用可能没有真正创建视觉元素，而是在内部维护了一个虚拟列表。删除时 `annotations.remove()` 可能只是从内部列表移除了记录，但没有触发视图更新。

**保留火种**：
- `rendition.annotations.highlight(cfiRange, { fill }, callback)` - 添加高亮的 API
- `rendition.annotations.remove(cfiRange)` - 删除高亮的 API
- `rendition.themes.default('.epubjs-hl', {...})` - 样式配置

**保留火种（可复用代码）**：

```typescript
// 清除所有高亮的正确方式
rendition.annotations.remove('highlight');

// 应用单个高亮
rendition.annotations.highlight(cfiRange, { fill: color }, () => {});
```
