# 任务闭环验证规范

## 核心原则

**任务完成 ≠ 代码写完**

真正的完成需要四个步骤：验证 → 文档 → 提交 → 确认

---

## 四大铁律

### 1. 立即验证

每个任务完成后立即验证，禁止批量标记：

```
□ npm run build 构建成功
□ npm run dev 启动无报错
□ 浏览器打开，控制台无错误
□ 功能实际可用
□ npm run check:lines 文件行数检查通过（所有文件<100 行）
```

**文件行数验证（新增）**：
- 创建新组件后，立即检查行数是否<100
- 超过 100 行必须按三层架构拆分
- 运行 `npm run check:lines` 自动化检查

### 2. 文档同步

文档必须描述**当前实际状态**，变更时同步更新：

- 代码变更 → 立即更新相关文档
- 任务状态 → 验证通过后再标记
- 目录结构 → 与实际文件保持一致

### 3. 引用验证

新组件创建后立即在某个页面中引用验证：

```
创建组件 → import 引用 → JSX 使用 → 验证渲染
```

禁止创建"孤儿组件"（存在但从未被引用）。

### 4. 立即提交

开发完成后立即提交，避免游离文件丢失：

```powershell
git add .
git commit -m "描述"
git status  # 确认工作区干净
```

---

## 闭环检查命令

```powershell
# 检查未跟踪文件（应为空）
git ls-files --others --exclude-standard

# 检查工作区状态
git status --short
```

## 失败案例

**本次功能消失问题的根因**：

| 问题 | 违反的铁律 |
|------|------------|
| 提示词组件未被引用 | 新组件未引用验证 |
| 重构计划标记完成但未执行 | 批量标记未验证 |
| 文档描述理想结构 | 文档与实际不同步 |
| 文件未提交 Git | 未立即提交 |

---

## 适用场景

- 创建新组件后
- 修改现有功能后
- 重构代码后
- 修复 Bug 后

---

## 连续失败复盘机制

### 触发条件

**连续 3 次修复失败** → 必须触发复盘

### 强制流程

```
用户反馈失败 → 计数 +1 → 达到 3 次 → 停止尝试 → 调用"失败复盘存档"技能
```

### 禁止行为

- 禁止在第 3 次失败后继续尝试新方案
- 禁止忽略用户反馈的失败
- 禁止依赖"上次方案应该能行"的侥幸心理

### 失败案例

**EPUB 导入点击无反应问题**：

| 次数 | 方案 | 结果 |
|------|------|------|
| 1 | pointerEvents: 'none' | 失败 |
| 2 | zIndex: 1 | 失败 |
| 3 | onClick + ref.click() | 失败 |
| 4 | label 包裹 input | 失败 |
| 5 | input 移到 div 外 | 失败 |
| 6 | label htmlFor + input id | 成功 |

**问题**：连续失败 5 次才成功，未在第 3 次失败时触发复盘

**根因**：Playwright 环境正常，Chrome 用户环境异常，环境差异导致误判

**教训**：用户环境验证 > 自动化测试验证
